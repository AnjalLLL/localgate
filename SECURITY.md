# Security Policy

## Reporting a vulnerability

**Please don't open a public issue.** Email **anjalghimire890@gmail.com** with:

1. What the vulnerability is
2. How to reproduce it
3. What an attacker could do with it
4. A suggested fix, if you have one

You'll get an acknowledgement within 48 hours. I'll work with you on a fix and coordinate
disclosure before anything is made public.

## How localgate handles secrets

### API keys

Keys are 32 bytes from `secrets.token_urlsafe` (256 bits of entropy) and stored as a
**SHA-256 hash**. The raw key is shown exactly once, at creation, and is not recoverable.

**SHA-256, not bcrypt — and that is deliberate.** Bcrypt is slow on purpose, because
passwords are low-entropy and an attacker runs a dictionary against them. An API key here
has no dictionary to try: brute force is infeasible regardless of hash speed. Meanwhile the
key is verified on *every request*, so a slow hash would add ~100ms of CPU to the hot path
and hand any unauthenticated client an easy denial-of-service — spray invalid keys, watch
the gateway burn CPU hashing them. Fast hashing of a high-entropy secret is the correct
construction; bcrypt exists for the opposite case. The full reasoning is in
[ADR 0003](docs/decisions/0003-sha256-for-api-keys.md).

This depends on keys being **generated by localgate, never chosen by a user** — `generate_key()`
is the only way one is created, and no endpoint accepts a user-supplied key.

### The admin key

Compared with `hmac.compare_digest`, not `==`. String comparison short-circuits at the first
differing byte, so how long it takes to fail leaks how much of the secret the caller got
right — enough, over many requests, to reconstruct it a byte at a time.

localgate **refuses to start** with `LOCALGATE_ENVIRONMENT=production` and the placeholder
admin key still in place. In development it starts, but warns at startup and reports it in
`GET /health`.

### What isn't logged

Prompts and completions are never written to logs. Sessions and keys appear by id only.
Database credentials are redacted in `GET /admin/config` and in startup logs.

`GET /admin/export` excludes key hashes.

## Isolation guarantees

- **Memory is scoped to a session, and sessions are owned by keys.** Retrieval cannot reach
  across sessions. Cross-session recall would be a data leak dressed up as a feature.
- **Reading another key's conversation returns 404, not 403.** 403 would confirm the session
  exists and let a caller enumerate other keys' session ids by probing.
- **Retrieved memory is framed as context, not instruction.** It's injected in a labelled
  system message that explicitly denies it instruction status, and always *after* the
  caller's own system prompt. Without that, anything a user once typed could arrive on a
  later turn wearing system authority.
- **Revocation is immediate.** A revoked key fails the auth lookup on its next request.

## Things you should know before deploying

- **`/metrics` is unauthenticated.** It exposes counts and latencies — never prompts or key
  material — but keep it off a public listener. Prometheus has no good way to hold a bearer
  token per target, which is why it's open.
- **Rate limits are per process.** N workers means the effective limit is N × what you
  configured. See [Deployment](docs/deployment.md).
- **The admin key is a single shared secret.** There are no admin roles or per-admin keys.
  Treat it like a root password.

## Practices

- All SQL goes through SQLAlchemy's parameterized queries. No string-built SQL anywhere.
- All request bodies are validated by Pydantic before any handler touches them.
- `tests/integration/test_key_management.py::test_every_route_is_guarded` asserts that no
  route is reachable without a key, except a short explicit list of endpoints meant to be
  open (`/health`, `/health/live`, `/metrics`, the docs).
